# Mobilise Robotics — Claude Code Routines Pack

This pack wires up five Claude Code routines for the Mobilise Robotics website (MALL × Keenon Robotics partnership site):

1. `fix-accessibility` — scheduled weekly. Auto-merge on clean CI.
2. `fix-security` — scheduled nightly. Auto-merge on clean CI.
3. `refresh-content` — scheduled monthly. Auto-merge on clean CI.
4. `site-review-and-ship` — combined weekly. Auto-merge on clean CI.
5. `publish-industry-blog` — scheduled weekly. **PR-only, human review required.**

## Setup sequence — do these in order

### Step 1 — drop the files into your repo

```bash
# from your Mobilise Robotics site repo root:
cp -r <path-to-this-pack>/docs .
cp -r <path-to-this-pack>/public .        # may merge with your existing public/
cp -r <path-to-this-pack>/.github .       # may merge with existing workflows
cp -r <path-to-this-pack>/routine-prompts .

# commit the scaffolding
git checkout -b setup/claude-routines-scaffold
git add docs/routines public/llms.txt .github/CODEOWNERS routine-prompts
git commit -m "chore: add Claude Code routines scaffolding"
git push origin setup/claude-routines-scaffold
# open PR, review, merge to main
```

### Step 2 — fill in the two template files

These are placeholders with `TODO:` markers. Routines will read them and make decisions based on what they contain, so gaps here = wrong automation.

- `docs/routines/BRD.md` — replace every `TODO:` with the real answer. 30-minute job.
- `docs/routines/BRAND.md` — replace every `TODO:` with approved brand facts. 30-minute job.

Commit these as a separate PR so the history shows the BRD is yours, not autogenerated.

### Step 3 — replace placeholder handles in CODEOWNERS

Open `.github/CODEOWNERS` and replace:
- `@ashish` → your actual GitHub handle.
- `@mobilise-robotics/content` → your org's content team slug, or drop it.

### Step 4 — set up branch protection on `main`

In GitHub → Settings → Branches → `main`:

- Require pull request before merging: **on**
- Require status checks to pass: add your build, test, lint jobs. Also add `lighthouse-ci`, `axe-ci`, `security-audit` once you create those jobs (step 5).
- Require review from Code Owners: **on** (this activates the CODEOWNERS file).
- Require branches to be up to date: **on**
- Include administrators: **on** — so no routine can bypass.

### Step 5 — add the CI jobs the routines depend on

Required status checks the routines expect:

- `build` — compiles / type-checks.
- `test` — unit + e2e.
- `lint` — eslint + stylelint.
- `axe-ci` — `@axe-core/cli` against the PR preview URL, fails on serious/critical violations.
- `lighthouse-ci` — asserts a11y ≥ 95, performance ≥ 80, SEO ≥ 95, best-practices ≥ 95.
- `security-audit` — `npm audit --audit-level=high` must pass; secret scan (gitleaks or trufflehog).
- `link-check` — no broken internal or external links in the PR's content files.
- `banned-phrases` — blocks blog PRs that contain any phrase from `BLOG-STYLE.md` §2.
- `readability-check` — Flesch ≥ 55 for blog PRs.
- `ai-detection-check` — < 60% AI-detected for blog PRs (directional).

If you don't have these workflows yet, start with the first four (`build`, `test`, `lint`, `axe-ci`). Add the others as you go; routines degrade gracefully when a check is missing (they still open PRs for human review).

### Step 6 — wire up lead-gen plumbing (before enabling `publish-industry-blog`)

The blog routine inserts CTAs that go nowhere if these aren't live:

- Form → CRM integration (Zoho / HubSpot / Salesforce / custom). Every site form submission reaches your CRM.
- WhatsApp Business click-to-chat link on every page.
- Calendly (or equivalent) for demo bookings.
- Google Analytics 4 + Google Search Console verified.
- Meta Pixel + LinkedIn Insight Tag installed.
- `robots.txt` reviewed — confirm AI crawlers (`GPTBot`, `ChatGPT-User`, `PerplexityBot`, `Google-Extended`, `ClaudeBot`, `anthropic-ai`, `CCBot`) are **allowed**. If you're on Cloudflare, the AI-bot control toggle is on by default and will block them — turn it off if you want AI-search visibility.

### Step 7 — create the routines in Claude Code

For each of the five prompts in `routine-prompts/`:

1. Open Claude Code → Routines → New routine.
2. Name: match the filename (e.g., `fix-accessibility`).
3. Prompt: paste the contents of the corresponding `.txt` file.
4. Repository: your Mobilise Robotics site repo.
5. Connectors:
   - All routines: GitHub, Web search.
   - `refresh-content` + `publish-industry-blog` + `site-review-and-ship`: also Google Drive (if BRD/brand docs live there), Slack.
6. Trigger — use these schedules (IST):

| Routine | Trigger | Notes |
|---------|---------|-------|
| `fix-accessibility` | `0 2 * * 1` (Mon 02:00) + API | Weekly |
| `fix-security` | `30 2 * * *` (nightly 02:30) + API | Nightly + Dependabot webhook |
| `refresh-content` | `0 6 1 * *` (1st of month, 06:00) + API | Monthly |
| `site-review-and-ship` | `0 3 * * 0` (Sun 03:00) + API | Weekly — don't enable until specialists are stable |
| `publish-industry-blog` | `0 5 * * 2` (Tue 05:00) + API | Weekly — PR opens in your morning |

### Step 8 — smoke test on a sandbox branch

Before pointing any routine at `main`:

1. Create a test branch: `git checkout -b routine-dryrun && git push`.
2. In the routine config, target the sandbox branch only.
3. Fire each routine manually via the API trigger.
4. Inspect what it wants to do. Look for: out-of-scope file touches, hallucinated facts in content PRs, dependency bumps you don't want.
5. Once the dry-run is clean, switch the routine to target `main` and enable the schedule.

### Step 9 — produce the top 3 lead magnets

The blog routine links to industry-specific PDFs. Until those exist, every matching post ships with an `[ASSET MISSING]` flag and a dead CTA.

Top 3 by likely sales pipeline weight:
1. Hotel robotics ROI worksheet (PDF).
2. Hospital delivery robot pilot checklist (PDF).
3. Mall cleaning robot RFP template (PDF).

Commission or draft these before the first blog ships. See `docs/routines/BLOG-ASSETS.md` for the full list.

### Step 10 — enable, one routine at a time

Do not enable all five on day one. Staggered rollout:

1. **Week 1**: enable `fix-accessibility` only. Watch it for a week.
2. **Week 2**: enable `fix-security`. Keep a11y running.
3. **Week 3**: enable `refresh-content`. Three specialists live.
4. **Week 4–6**: enable `publish-industry-blog`. Review first 2–3 PRs closely.
5. **Week 7+**: once the specialists are all stable and producing clean runs, optionally enable `site-review-and-ship`. Many teams never enable this and stay on the three specialists; that's fine.

## Honest expectations

- **Time to first published post**: ~2 weeks (after BRD/brand are filled in and CI is wired).
- **Time to ranking movement**: 3–6 months of consistent weekly publishing, plus backlink work.
- **Time to meaningful AI-citation presence**: 60–90 days after consistent publishing + `llms.txt` live + AI crawlers allowed.
- **What the routines guarantee**: well-researched, on-brand, technically optimized content shipping reliably.
- **What the routines cannot guarantee**: position-1 rankings, AI citations on any specific prompt, lead volume targets. Those depend on execution beyond the routines (PR, backlinks, real case studies, photography, brand-building).
